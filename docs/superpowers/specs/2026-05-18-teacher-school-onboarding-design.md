## Problem

A first-time visitor ‚Äî teacher, school administrator, or student ‚Äî cannot complete a seamless onboarding today. Five concrete frictions:

1. **`/auth` is a single page** for both login and signup. New users see signup mixed with login, and the role they intend to play is never asked. Routing after signup is heuristic and breaks for users with no memberships.
2. **School creation is admin-gated** (Lingual approves `school_requests`), but the wait is opaque. There is no email notification to Lingual admins, no status page rich enough to keep the admin engaged, and no obvious next step.
3. **The school-request form is too thin** (name, type, website, Canvas URL). It collects neither the data needed to verify the org nor the operational context (school type, location, languages taught) that downstream features depend on.
4. **Teachers cannot self-serve discover an existing school.** The only path is an invite code generated by an admin ‚Äî and that code now auto-approves (`commit 4bbcbe3`), removing the human-in-the-loop step the admin actually wants for known-bad-actor defense.
5. **Lingual admin tooling stops at request approval.** There is no operational view of active organizations, no way to suspend a misbehaving org, no audit trail of vendor actions on customer data. SOC 2 and FERPA both expect more.

## Goals

1. Split signup and login into two purpose-built pages; make signup role-aware from the first click.
2. Collect a rich, audit-grade payload at school-registration time via a 4-step wizard, with autosave and resume-on-return.
3. Give teachers two paths to join an existing org ‚Äî invite code (primary) or name search (fallback) ‚Äî both gated by a one-click admin approval.
4. Build an outbox-pattern email system using Resend so every state transition (request submitted, approved, declined, suspended) notifies the right party.
5. Expand the Lingual admin panel from request-only to a full org-management surface: dashboard, requests, organizations list, org detail (overview / members / classes / audit), suspend/restore ‚Äî with strict FERPA boundaries.
6. Migrate existing users without memberships via a one-time backfill plus a lazy role-pick modal on next login.

## Non-goals

- **Self-serve school creation without Lingual approval.** The admin gate stays in place ‚Äî we accelerate the wait, not remove it. K-12 fraud risk is unchanged.
- **Multi-role users.** A user is one of {student, teacher, school_admin} at a time. Multi-role (e.g., a teacher who also wants to practice as a learner) is deferred to v1.5.
- **Org deletion via the panel.** Suspend (reversible) is in scope. Hard delete is rare and is handled by direct Firestore operations until a v1.5 review.
- **LMS integrations beyond Canvas.** Google Classroom and Schoology placeholders only.
- **Parent/guardian as a signup role.** Guardians continue to use the existing secure-link `guardian_packets/` flow.
- **District-admin role (above school admin).** Deferred until at least one pilot customer has the structure.
- **Realtime listeners for status changes.** Polling every 30s is acceptable for v1.

## Core invariants

1. **Role pick ‚âÝ role grant.** Selecting "Admin" on signup writes `intended_role` to the user's profile but does not create any membership. Memberships only come from (a) Lingual approving a school request, (b) a school admin approving a teacher join request, or (c) a student enrolling in a class via join code or LTI launch.
2. **Org existence requires Lingual approval.** No path creates an `organizations/` doc without a corresponding approved `school_requests/` doc.
3. **No student PII in the Lingual admin panel.** Lingual staff see staff (school admins + teachers) names/emails and aggregate counts only. Student-level data is never exposed in the panel.
4. **Every Lingual admin action is audited** in `lingual_admin_audit/`, including org detail views.
5. **Email send is decoupled from business transactions** via the Firestore outbox pattern. A Resend failure never breaks a business transaction.

## Personas

| Persona | First-time goal | Outcome of new flow |
|---|---|---|
| **Student** | Practice speaking, possibly via a teacher's class | Lands on `/app/learn` within 60 seconds; can paste a class join code if they have one |
| **Teacher (school on Lingual)** | Join their existing school, then build assignments | Picks role ‚Üí creates account ‚Üí enters invite code or searches ‚Üí admin approves (often within an hour) ‚Üí lands on `/app/teacher` |
| **Teacher (school not on Lingual)** | Bring Lingual to their school | Picks Teacher ‚Üí search returns nothing ‚Üí "Can't find my school" ‚Üí optional pivot to Admin wizard |
| **School Administrator** | Register the school and onboard their teachers | Picks Admin ‚Üí 4-step wizard ‚Üí submits ‚Üí pending page shows estimated review time and lets them pre-list teachers ‚Üí Lingual approves within 24h ‚Üí org and admin membership are created, pre-invites are sent, admin lands on `/app/admin` |
| **Lingual Admin** | Operate the platform safely | Sees pending requests, can approve/decline with reason, sees active orgs, can suspend a misbehaving org, never sees student PII |

## Section 1 ‚Äî Routing & Page Split

**Current state**: `/auth` serves login + signup + reset. `/school/setup` is an admin-only one-page form. New users get bounced around by heuristics in `homeRoutes.ts`.

**Target routes**:

| Route | Purpose | Auth |
|---|---|---|
| `/` | Landing, role-aware CTAs | public |
| `/login` | Existing account sign-in (+ password reset link) | public |
| `/signup` | Step 1: role pick + Step 2: account creation | public |
| `/signup/student/setup` | Step 3 ‚Äî student | signed in |
| `/signup/teacher/join-org` | Step 3 ‚Äî teacher (code or search) | signed in, no memberships |
| `/signup/teacher/pending` | Awaiting school admin approval | signed in |
| `/signup/admin/org-wizard` | Step 3 ‚Äî admin (4-step wizard) | signed in, no memberships |
| `/signup/admin/pending` | Awaiting Lingual approval | signed in |
| `/app/learn` | Student dashboard | student or legacy learner |
| `/app/teacher` | Teacher dashboard | teacher membership |
| `/app/admin` | School-admin dashboard | school_admin membership |
| `/app/lingual-admin/*` | Lingual admin panel (Section 6) | lingual_admin membership |
| `/auth` | Permanent redirect ‚Üí `/login` | ‚Äî |
| `/school/setup` | Permanent redirect ‚Üí `/signup/admin/org-wizard` | ‚Äî |

**Onboarding state cursor**: `users/{uid}/profile.onboarding_state` enum:

| Value | Meaning | Resume target |
|---|---|---|
| _missing_ + memberships=0 + legacy | Legacy user, role not picked | trigger role-pick modal |
| `role_selected` | Step 2 just completed, intended_role written | Step 3 for that role |
| `student_setup` | Student Step 3 in progress | `/signup/student/setup` |
| `teacher_pending` | Teacher join request submitted, awaiting admin | `/signup/teacher/pending` |
| `org_creation_pending` | Admin wizard in progress; substate in draft | `/signup/admin/org-wizard?step=<n>` |
| `awaiting_lingual` | Admin wizard submitted, awaiting Lingual | `/signup/admin/pending` |
| `complete` | Active membership exists | role-specific dashboard |

**Role-aware dispatcher** (`homeRoutes.ts` rewrite):

```text
if user has lingual_admin membership      ‚Üí /app/lingual-admin
elif user has active school_admin mem.    ‚Üí /app/admin
elif user has active teacher membership   ‚Üí /app/teacher
elif user has active student enrollment   ‚Üí /app/learn
elif onboarding_state == 'complete'       ‚Üí /app/learn  (legacy learner)
elif onboarding_state present             ‚Üí resume target (see table above)
else                                       ‚Üí /signup  (role picker, may show legacy modal)
```

**Implementation footprint**:
- Split `frontend/src/pages/AuthPage.tsx` (417 lines) into `LoginPage.tsx` and the new `SignupPage.tsx` shell.
- Rewrite `frontend/src/lib/homeRoutes.ts` as a small dispatcher driven by membership + `onboarding_state`.
- Add routes to `frontend/src/App.tsx`.
- New components: `RolePicker`, `AccountCreator`, `SignupShell`, `SignupBreadcrumb`.
- Add legacy redirects for `/auth` and `/school/setup`.

## Section 2 ‚Äî Role-aware Signup Flow

**Order**: Role pick ‚Üí Account creation ‚Üí Role-specific step.

### Step 1: Role picker (`/signup`)

Three role cards with copy:
- **Student** ‚Äî "Practice speaking. Join your teacher's class."
- **Teacher** ‚Äî "Manage classes and assignments. Join your school."
- **School Administrator** ‚Äî "Register your school. Manage teachers and compliance."

`?role=student|teacher|admin` query param pre-selects from landing-page CTAs. Selection lives in local state only; Firebase is not touched until Step 2. Bottom link: "Already have an account? Log in ‚Üí" routes to `/login`.

### Step 2: Account creation

Two paths:
- **Continue with Google** ‚Äî `signInWithPopup` (reuses existing AuthPage logic).
- **Sign up with email** ‚Äî full name, email, password, confirm password.

The role chosen in Step 1 is held in component state and POSTed alongside the Firebase ID token.

**Backend contract change** ‚Äî `POST /api/auth/verify` now accepts an optional `intended_role`:

```json
{
  "idToken": "...",
  "intended_role": "student" | "teacher" | "admin"
}
```

Behavior:
- If this is the user's first verify and `intended_role` is provided, write it to `users/{uid}/profile.intended_role` and set `onboarding_state = 'role_selected'`.
- If the user already has active memberships, ignore `intended_role` (existing memberships win ‚Äî prevents role spoofing for returning users).
- Otherwise, return the normal payload plus the resume cursor.

### Step 3: Dispatch

After verify, the frontend routes based on the response payload:

| Response | Route |
|---|---|
| Active memberships present | role-aware dispatcher (Section 1) |
| `intended_role=student`, no memberships | `/signup/student/setup` |
| `intended_role=teacher`, no memberships | `/signup/teacher/join-org` |
| `intended_role=admin`, no memberships | `/signup/admin/org-wizard` |
| No `intended_role` and not legacy | `/signup` (force role pick) |
| Legacy user (Section 7) | role-pick modal mounts in place |

### Edge cases

- **Google account already exists with a different role**: response includes existing memberships; `intended_role` is ignored and the user goes to their existing dashboard.
- **Mid-flow role change**: each Step 3 page has a "‚Üê Change role" link that clears the in-progress draft and returns to Step 1. New role updates `intended_role`.
- **Bail after Step 2**: `onboarding_state='role_selected'` persists. Next login resumes at Step 3 for that role.

## Section 3 ‚Äî Teacher Join-Org Flow

Hybrid: invite code (primary) + name search (fallback). Both paths require admin approval.

### UI panes on `/signup/teacher/join-org`

**Pane A ‚Äî Entry**: "Do you have an invite code?" with two large options leading to Pane B or Pane C.

**Pane B ‚Äî Code entry**: text input for the code (e.g., `XYZ-A1B2-C3`). Backend validates code existence, expiry, revocation status, and remaining uses. On success, creates a `teacher_join_requests/{id}` with `source='invite_code'` and `invite_code_id` set, then redirects to the pending page.

**Pane C ‚Äî Search**: debounced name search against `GET /api/organizations/search?q=<name>`. Results show name, city/state, school_type only (no sensitive metadata). Click a result ‚Üí confirm dialog ("Request to join [Org]?") ‚Üí submit ‚Üí request created with `source='search'` and no `invite_code_id`.

**"Can't find my school" link** offers three options:
1. "I'm actually an administrator" ‚Üí routes to `/signup/admin/org-wizard` (updates `intended_role` to `admin`).
2. "Try a different spelling" ‚Üí returns to search.
3. "Contact us" ‚Üí mailto/help link.

### Pending page (`/signup/teacher/pending`)

Shows: target org name, status (pending / approved / declined), submitted timestamp. Polls `GET /api/teacher-join-requests/me` every 30 seconds. If approved, redirects to `/app/teacher`. If declined, displays the decline reason and offers retry/support. Includes a "Cancel request" action.

### Admin-side review

A new section on `TeacherDashboardPage` (visible to school_admins of the org): **"Pending teacher requests (N)"**. Each row shows requester name, email, source, requested time. Inline Approve / Decline buttons. Decline opens a modal with a required reason and category. Approve creates a `memberships/{id}` (role: `teacher`, status: `active`) and triggers the approval email.

### Data model

**Existing `teacher_invitations`**: unchanged; continues to store admin-generated codes. Auto-approve behavior from `commit 4bbcbe3` is **removed** ‚Äî code use now creates a request, not an immediate membership.

**New `teacher_join_requests/{request_id}`**:

| Field | Type | Notes |
|---|---|---|
| `request_id` | str | doc id |
| `uid` | str | requester |
| `org_id` | str | target org |
| `source` | `'invite_code'` \| `'search'` | |
| `invite_code_id` | str? | populated if `source='invite_code'` |
| `status` | `'pending'` \| `'approved'` \| `'declined'` \| `'cancelled'` | |
| `requested_at` | timestamp | |
| `reviewed_at` | timestamp? | |
| `reviewed_by_uid` | str? | admin who reviewed |
| `decline_reason` | str? | |

**Firestore rules**: requester can read their own doc; school_admins of `org_id` can read/update docs in their org.

### Endpoints

| Method | Path | Purpose | Auth |
|---|---|---|---|
| GET | `/api/organizations/search?q=` | rate-limited search (5 req/sec/user) | signed-in user |
| POST | `/api/teacher-join-requests` | submit `{ code }` or `{ org_id }` | signed-in user with no `teacher` membership in target org |
| GET | `/api/teacher-join-requests/me` | poll my pending request | requester |
| DELETE | `/api/teacher-join-requests/me` | cancel my pending request | requester |
| GET | `/api/teacher-join-requests?org_id=` | admin lists their org's pending | school_admin |
| POST | `/api/teacher-join-requests/<id>/approve` | approve | school_admin |
| POST | `/api/teacher-join-requests/<id>/decline` | decline with reason | school_admin |

### Edge cases

- **Already member of a different org**: 422 "You're already a member of [Org]. Contact support to change." Multi-org membership is out of scope for v1.
- **Pending request already exists**: enforce one active pending request per user; new submissions return 409 unless the existing one is cancelled first.
- **No admin response after 7 days**: outbox writes a reminder email to org admins (scheduled send).

## Section 4 ‚Äî Admin 4-Step Org Wizard

**URL**: `/signup/admin/org-wizard?step=N`. Step state is in the URL so refresh and back-button work safely.

### UX chrome

Top progress bar (4 dots), left sidebar listing the steps, right panel showing the active step's form, bottom controls: "Save & Continue", "‚Üê Back", "Save draft & exit". Draft autosave on every field blur to `school_creation_drafts/{uid}`. Resume from the last completed step on next login.

### Step 1 ‚Äî Organization identity (required)

| Field | Type | Validation |
|---|---|---|
| Organization name | text | required, 2‚Äì120 chars |
| Organization website | URL | required, valid URL |
| Country | dropdown (IP-detected default) | required |
| State / Province | cascading dropdown | required |
| County / District | text | optional outside US |
| School type | radio: Middle / High / K-12 / University / Language academy / District / Other | required |
| Public / Private | radio: Public / Private / Charter / Other | required |
| Grade size (students per grade level) | radio: `<50` / `50‚Äì100` / `100‚Äì200` / `200‚Äì500` / `500+` | required |
| Official email domain(s) | chip input (e.g., `@ssfs.org`) | optional, multi-value |

### Step 2 ‚Äî Admin identity and authorization (required)

| Field | Type | Validation |
|---|---|---|
| Your full name | text (prefilled from Firebase `displayName`) | required |
| Your school email | email (prefilled from Firebase auth email) | required; warns (not blocks) if it doesn't match Step 1 domains |
| Your role / title | dropdown + custom: Teacher / Department chair / Principal / Vice Principal / IT admin / LMS admin / Other | required |
| Authorization attestation | checkbox | required ‚Äî text below |

Attestation text: *"I confirm that I am authorized by [Org Name] to create and manage this organization on Lingual. I understand that misrepresentation may result in account termination and is logged for audit."*

On submit of the wizard, the attestation is stored with `confirmed_at`, `ip_hash` (SHA-256 of remote IP + a per-deploy salt), and `user_agent`.

### Step 3 ‚Äî Integration and curriculum (optional, skippable)

"Tell us how you teach. You can fill this in later from settings."

**Integration sub-section**: "Does your school use Canvas LMS?" Yes / No / Not sure. Yes ‚Üí Canvas instance URL + multi-select Integration types: LTI 1.3 assignment launch, Roster sync, Grade passback, SSO only. No / Not sure ‚Üí placeholder: "Google Classroom and Schoology support coming soon."

**Curriculum sub-section**:
- Target student age/grade range (multi-select chips: K-2 / 3-5 / 6-8 / 9-12 / Undergrad / Graduate / Adult Ed)
- Languages taught (multi-select chips, populated from `ALLOWED_LEARNING_LOCALES`)
- Course frameworks (multi-select: AP / ACTFL / CEFR / IB / School-specific / None)

### Step 4 ‚Äî Review and submit

Read-only summary grouped by section. "Edit" links jump back to the relevant step. Optional **pre-invite teachers** input: chip list of teacher emails. On approval, each pre-invited email gets a `teacher_invitations` record and a `teacher_invitation` outbox email.

Submit triggers:
1. `POST /api/school-requests` with the full enriched payload.
2. Backend creates `school_requests/{id}` with `status='pending'`.
3. Outbox: `school_request_to_lingual` email to all `lingual_admin` users.
4. `users/{uid}/profile.onboarding_state = 'awaiting_lingual'`.
5. Frontend redirects to `/signup/admin/pending`.

### Pending page (`/signup/admin/pending`)

Shows submission summary, estimated review time ("usually within 24 hours"), pre-invited teacher list (with edit / remove), and "Cancel request" / "Edit request" actions. Polls `GET /api/school-requests/me` every 30 seconds. On approval, surfaces a "Continue to dashboard" CTA. On decline, surfaces the reason and offers Resubmit / Contact support.

## Section 5 ‚Äî Email Outbox + Resend Integration

Pattern: business action writes to `outbox_emails/`. A Firestore-triggered Cloud Function picks up the document, renders the template, calls Resend, and writes back the result. Business transactions never await the outbound API.

### Data model

**`outbox_emails/{email_id}`**:

| Field | Type | Notes |
|---|---|---|
| `recipient` | `{ email, name? }` | |
| `template_id` | enum | see template table |
| `template_data` | object | merge data for the template |
| `status` | `pending` \| `sending` \| `sent` \| `failed` \| `dead_letter` \| `sent_dev` | |
| `scheduled_for` | timestamp | immediate or future (reminders) |
| `attempt_count` | int | for retry |
| `last_attempt_at` | timestamp? | |
| `sent_at` | timestamp? | |
| `error` | str? | |
| `resend_message_id` | str? | from Resend API |
| `created_at` | timestamp | |
| `created_by_uid` | str? | null for system events |
| `related_entity` | `{ type, id }`? | e.g., `{ type: 'teacher_join_request', id: '...' }` |

### Templates (v1)

| Template ID | Trigger | Recipient | Subject |
|---|---|---|---|
| `teacher_join_request_to_admin` | Teacher submits join request | school_admins of org | "New teacher request to join [Org]" |
| `teacher_join_approved` | Admin approves teacher join | Teacher | "Welcome to [Org] on Lingual" |
| `teacher_join_declined` | Admin declines | Teacher | "Your request to join [Org] was not approved" |
| `school_request_to_lingual` | Admin submits school registration | All `lingual_admin` users | "New school registration: [Org]" |
| `school_request_approved` | Lingual approves school | Requester | "Your school [Org] is now on Lingual" |
| `school_request_declined` | Lingual declines | Requester | "Your school registration needs more info" |
| `teacher_invitation` | School admin invites (manual or pre-invite list) | Invitee email | "[Org] is inviting you to teach on Lingual" |
| `org_suspended` | Lingual suspends org | school_admins of org | "[Org] has been temporarily suspended" |
| `org_restored` | Lingual restores org | school_admins of org | "Your access to Lingual has been restored" |
| `join_request_reminder_to_admin` | 7 days no review | school_admins | "Reminder: teacher request awaiting your review" |
| `school_request_reminder_to_lingual` | 3 days no review | lingual_admins | "Reminder: school request awaiting review" |

Templates rendered with [React Email](https://react.email/) for component reuse and inline-CSS conversion.

### Cloud Function

`functions/sendOutboxEmail.ts`:
- `onDocumentCreated('outbox_emails/{emailId}')`: render template, call `resend.emails.send`, update doc status. Retry with exponential backoff; mark `dead_letter` after 5 attempts.
- `onSchedule('every 5 minutes')`: pick up `status='failed'` with `attempt_count < 5`, and any `status='pending'` with `scheduled_for <= now` (handles future-scheduled reminders).

Dev mode: when `RESEND_API_KEY` is unset, the function logs the rendered email and marks the doc `sent_dev` without calling Resend.

### Environment

- New env: `RESEND_API_KEY` (only in Cloud Function runtime).
- Domain verification for `lingual.app` via Cloud DNS (SPF, DKIM, DMARC records).
- `from`: `Lingual <noreply@lingual.app>`.

### Backend integration

Backends write outbox docs in the same transaction as the triggering business write. Example:

```python
# backend/routes/teacher_requests.py
with db.transaction() as tx:
    request_ref = db.collection('teacher_join_requests').document()
    tx.set(request_ref, {...})

    for admin in list_school_admins(org_id):
        tx.set(db.collection('outbox_emails').document(), {
            'recipient': {'email': admin['email'], 'name': admin['display_name']},
            'template_id': 'teacher_join_request_to_admin',
            'template_data': {...},
            'status': 'pending',
            'scheduled_for': SERVER_TIMESTAMP,
            'attempt_count': 0,
            'created_at': SERVER_TIMESTAMP,
            'related_entity': {'type': 'teacher_join_request', 'id': request_ref.id},
        })
```

### `firestore.rules`

`outbox_emails`: service-account writes only (backend admin SDK); no client read or write. Cloud Function admin SDK bypasses rules and operates as needed.

### Testing

- Unit: template render snapshot tests.
- Integration: backend writes outbox doc with expected fields (Firestore emulator).
- E2E: Cloud Function emulator + mocked Resend client; verify status transitions.

## Section 7 ‚Äî Legacy User Lazy Migration

Two layers: a one-time backfill plus a lazy modal for the remainder.

### Backfill (run before frontend modal rollout)

`scripts/backfill_legacy_user_roles.py`:

For every user without `intended_role` set, look at active memberships in priority order:
- `school_admin` membership ‚Üí `intended_role='admin'`, `onboarding_state='complete'`.
- `teacher` membership ‚Üí `intended_role='teacher'`, `onboarding_state='complete'`.
- `student` enrollment ‚Üí `intended_role='student'`, `onboarding_state='complete'`.
- Otherwise: do not modify. The lazy modal handles these.

Includes `--dry-run` mode. Logs counts by transition.

### Lazy modal

`/api/auth/verify` adds a flag `requires_legacy_role_pick: bool`, true iff:
- `intended_role` missing AND
- `onboarding_state` missing AND
- No active memberships.

When true, the frontend mounts `LegacyRoleMigrationModal` (blocking ‚Äî no close, no escape). Body copy:

> Welcome back!
> Lingual now supports classrooms.
> How are you using Lingual?
> [Student] [Teacher] [School Administrator]
> Your existing progress stays with you.

On pick ‚Üí `POST /api/auth/migrate-role { role }`:
- Validate role; server-side recheck that the user is legacy (defense-in-depth).
- Student ‚Üí `intended_role='student'`, `onboarding_state='complete'`. Modal closes, current page retained.
- Teacher ‚Üí `intended_role='teacher'`, `onboarding_state='role_selected'`. Frontend navigates to `/signup/teacher/join-org`.
- Admin ‚Üí `intended_role='admin'`, `onboarding_state='role_selected'`. Frontend navigates to `/signup/admin/org-wizard`.

Telemetry: log `legacy_role_pick { role, uid, ts }` for product distribution insight.

### Data preservation invariant

Migration never deletes or mutates existing learner data (assessment, results, chats, profile). Role context is layered on top. A student-picked legacy user continues with their existing flow; a teacher- or admin-picked legacy user keeps their learner data dormant for a future "switch to learning mode" (v1.5).

### Backend additions

- `backend/routes/auth.py`: extend `/api/auth/verify` response with `requires_legacy_role_pick`; add `POST /api/auth/migrate-role`.
- `database.py`: `mark_user_legacy_role_picked(uid, role)`.

### Frontend additions

- `components/LegacyRoleMigrationModal.tsx`.
- `AuthProvider`: detect `requires_legacy_role_pick` and mount the modal.
- `api/auth.ts`: `migrateRole` function.

### Rollout order

1. Land backend `requires_legacy_role_pick` flag and `migrate-role` endpoint (modal component absent ‚Üí no behavior change).
2. Run backfill script on staging (dry run, then real).
3. Run backfill on production.
4. Land the frontend modal.
5. Monitor telemetry for 1 week: modal-impression count, role distribution.
